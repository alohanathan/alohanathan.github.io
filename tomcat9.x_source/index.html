2.1 Tomcat是什么

2.1.1 Tomcat介绍

​ Tomcat是Apache 软件基金会(Apache Software Foundation)的Jakarta 项目中的一个核心项目（2011年12月被独立出去，成为Apache顶级项目），由Apache、Sun 和其他一些公司及个人共同开发而成。由于有了Sun 的参与和支持，最新的Servlet 和JSP 规范总是能在Tomcat 中得到体现，Tomcat 9支持最新的Servlet 4.0 和JSP 2.3 规范。因为Tomcat 技术先进、性能稳定，而且免费，因而深受Java 爱好者的喜爱并得到了部分软件开发商的认可，成为比较流行的Web 应用服务器。

　　Tomcat 服务器是一个免费的开放源代码的Web 应用服务器，属于轻量级应用服务器，在中小型系统和并发访问用户不是很多的场合下被普遍使用，是开发和调试JSP 程序的首选。对于一个初学者来说，可以这样认为，当在一台机器上配置好Apache 服务器，可利用它响应HTML(标准通用标记语言下的一个应用)页面的访问请求。实际上Tomcat是Apache 服务器的扩展，但运行时它是独立运行的，所以当你运行tomcat 时，它实际上作为一个与Apache 独立的进程单独运行的。

　　当配置正确时，Apache 为HTML页面服务，而Tomcat 实际上运行JSP 页面和Servlet。另外，Tomcat和IIS等Web服务器一样，具有处理HTML页面的功能，另外它还是一个Servlet和JSP容器，独立的Servlet容器是Tomcat的默认模式。不过，Tomcat处理静态HTML的能力不如Apache服务器。Tomcat最新版本为10.1.4（GitHub - 2022年12月06日上传的Tag包）。

Note（Apache）:

上世纪80年代，互联网在美国大学开始流行，这就促使了一帮志同道合的人开始编写基于HTTP通信协议的服务器端和客户端软件。客户端软件最后演变成了Mozilla Firefox，服务端软件是最早期的Web服务器软件之一，名叫NCSA HTTPd，NCSA是美国计算机名校伊利诺大学香槟学校的国家超级计算机应用中心（National Center for Supercomputing Applications），由于这个HTTPd仅仅是为了学术研究，后面实验成功后就没有再深入下去了，成员就随之解散了，但好在项目是开源出去了。
后面随着互联网对于HTTP服务器软件的需求越来越大，HTTPd公开的源码被大量人广泛下载和升级迭代，版本之多难以管理且BUG也是越改越多，为了解决各自为战单打独斗的的局面，8名志同道合的Developer一拍即合成立了一个小组，一起重整NCSA HTTPd程序，发布了一个基于NCSA HTTPd的可靠的服务器软件，他们将这个软件命名为“Apache”，全称Apache HTTP Server。
Apache本是美洲原住印第安人一支部落的名字，这个部落因为高超的作战策略和无穷的耐性而闻名，同时也是最后一个屈服于美国政府的名族。开发小组以它来命名寓意着Apache HTTP Server这款软件高效、可靠，同时表达了大公司迟早会参与竞争并"教化"这块最早的开源网络之地的担心。另外，由于整个软件是在NCSA HTTPd基础上打了很多补丁程序，因此他们又戏称它是"A Patchy /ˈpætʃi 零散的，补丁的/ Web Server",意为一个打了很多补丁的Web服务器软件。"A Patchy" 与 "Apache"谐音，故以Apache命名一语双关。
最后，为了Apache HTTP Server软件不屈服于商业公司的管控，且为了后续软件的更好发展，Apache开发小组在1999年6月成立了一个非盈利的基金会，它就是后来大名鼎鼎的 ASF（Apache Software Foundation）。

ASF孵化了很多顶级的项目，Tomcat就是其中之一，具体ASF有哪些项目，可以访问：Apache开源项目列表

Note（Tomcat）:

Tomcat之父詹姆斯•邓肯•戴维森（James Duncan Davidson）曾对网友提出的"Tomcat为什么叫Tomcat"这样回答：
"Tomcat 的诞生是为了响应对独立 Servlet 规范实现的需求。James 写这篇文章是希望它最终能够开源。由于大多数开源项目都有关于 O'reilly(一个出版公司，很出名，专注于开放系统图书类的) 的书籍，因此 James 认为这个项目应该以动物的名字命名。从本质上说，James 在想一种会出现在 O'reilly 书封面上的动物。James 想到了 Tomcat，因为 Tomcat 这种动物，代表着一种 能够自己照顾好自己的动物。"

Note（Jakarta）:

2017年8月，Oracle（甲骨文）决议将Java EE（Java Enterprise EdITion）移交给开源组织，最初Eclipse基金会接手。这应该是甲骨文完成对Java品牌控制的最新举措，虽然之前Sun的资产曾经被甲骨文围歼得只剩一个VirtualBox还能喘口气。不过，甲骨文可不允许开源组织用Java的名号，于是Eclipse选出了 "Jakarta EE" 和"Enterprise Profile"两个，最终前者以64.4%的票数获胜。也就是说，Java EE从此正式更名为Jakarta EE（雅加达）, 而Jakarta（为了发展Java Web容器而启动的Java项目，旗下囊括了众多基于Java语言的开源软件，如：Tomcat、Maven、JMeter、Structs等，后来这些子项目都独立出去成为ASF的顶级项目了）早就存在在了，它是Apache早期开源项目之一。

名词 描述
Server 即服务器，每个tomcat程序启动后，就是一个server。它内部包含多个Service，负责管理和启动各个Service，同时也负责监听Shutdown关闭容器命令。
Service Servlet容器服务，其将连接器和Servlet处理引擎结合在一起进行管理，同时其内部还维护了Executor线程池，负责执行Service组件周期性任务（默认10秒执行一次生命周期监听器）。Service内部包含多个Connector和一个Engine。
Connector 连接器用于处理连接和并发，内部屏蔽了不同应用层协议和网络IO模型。Tomcat支持的应用层协议有：①HTTP1.1 ②HTTP2（性能非常优越，浏览器请求并发数不受限制） ③AJP（Apache JServ Protocol，定向包协议，通过二进制进行传输而不是文本，因此性能高）；Tomcat支持的网络IO模型有：①BIO（同步阻塞，Tomcat8.5之后已经移除掉了）②NIO（同步非阻塞，基于JDK的NIO库实现，Tomcat内部实现了自己的Reactor模型，即单Reactor多线程，不要再说为什么Tomcat不用Netty了，因为Tomcat自己实现了啊）③AIO（异步非阻塞，体现在Tomcat中就是NIO2，NIO2和AIO是一回事，基于JDK的NIO2库实现）④APR（使用C++本地库实现，从操作系统层面来处理异步式IO，性能较高，但是需要安装依赖）。
Engine Servlet请求处理引擎，所有的请求都是通过Engine处理的。
Host 虚拟主机，用于进行请求的映射处理。每个虚拟主机可以看做独立的请求文件。
Realm 用于配置安全管理角色，通常读取tomcat-uesrs.xml进行验证。
Context 上下文，对应于Web应用。
Adapter 由于协议不同，客户端发过来的请求信息也不尽相同，Tomcat定义了自己的Request 类来封装这些请求信息。ProtocolHandler接口负责解析请求并生成Tomcat Request类。但是这个Request对象不是标准的ServletRequest，不能被CoyoteAdapter使用，这是设配器模式的经典运用，连接器调用CoyoteAdapter的Service方法，传入的是Tomcat Request对象，CoyoteAdapter负责将这个对象转换成ServletRequest，再传递给容器。
ProtocolHandler Coyote 协议接口，通过Endpoint和Processor，实现针对具体协议的处理能力。Tomcat按照协议和I/O提供了6个实现类：AjpNioProtocol，AjpAprProtocol，AjpNio2Protocol ， Http11NioProtocol ， Http11Nio2Protocol ，Http11AprProtocol
Processor 用于将Endpoint接收到的socket处理封装成request对象，实现HTTP协议。
Endpoint 通信端点，处理底层socket网络连接，实现TCP/IP协议。
Acceptor 循环监听和限制客户端请求，客户端请求超过最大连接数，就阻塞，否则将socket交给Poller处理，属于守护线程。
Poller 工作线程，tomcat处理客户端请求的工作模式类似于netty（Reactor模型，IO多路复用），将Acceptor扔过来的客户端socketChannel进行事件注册，注册后的事件称作PollerEvent，保存到队列中，poller内部维护一个selector，poller线程循环事件队列，并通过selector监听客户端通道IO读写事件，这个过程属于同步非阻塞，效率非常高，如果IO读写就绪，就将当前的channel交给SocketProcessor任务，该任务最终会扔给Tomcat扩展JDK的线程池来处理。同Acceptor，该线程也是一个守护线程。
SocketProcessor 处理有IO读写事件发生的客户端channel，然后封装一番交给对应的protocolHandler进行处理。
ThreadPoolExecutor 处理客户端请求的线程。

2.5 什么是Servlet

2.5.1 Servlet介绍

​ Servlet（Server Applet /ˈæplət/，服务端小程序，是服务端的一小部分），全称Java Servlet。是用Java编写的服务器端程序。其主要功能在于交互式地浏览和修改数据，生成动态Web内容。狭义的Servlet是指Java语言实现的一个接口（Servlet就是一群人来制定Java应用中使用Web时的各种规范，统一接口，其内部实现由各个厂商自己实现，Tomcat 、Jetty、 JBoss等等应运而生），广义的Servlet是指任何实现了这个Servlet接口的类，一般情况下，人们将Servlet理解为后者。

​ Servlet运行于支持Java的应用服务器中。从原理上讲，Servlet可以响应任何类型的请求，但绝大多数情况下Servlet只用来扩展基于HTTP协议的Web服务器。

​ 最早支持Servlet标准的是JavaSoft的Java Web Server，此后，一些其它的基于Java的Web服务器开始支持标准的Servlet。

​ 开发Servlet有三种方法，一种是实现servlet接口，一种是继承GernericServlet，一种是继承HttpServlet。 为什么会有三种方法呢？因为这个Servlet这个技术并不是说一出现之后就是成熟的，他经历过这三个发展阶段，因此它具有这三种方式，其实在最先前是实现Servlet接口方法来开发的，后面的时候技术人员发现这样写似乎不太方便，所以又发展出来一个叫做继承GenericServlet的方法来实现，后来又发现这个也不是很方便，所以又发展出来了继承 HttpServevlet方式来实现（设计模式：模板方法）。

package com.appleyk;

import javax.servlet.ServletException;
import javax.servlet.annotation.WebServlet;
import javax.servlet.http.HttpServlet;

/**
 * <p>用户Servlet，主要处理和用户信息相关的业务</p>
 *
 * @author Appleyk
 * @version v1.0
 * @blob https://blog.csdn.net/appleyk
 * @github https://github.com/kobeyk
 * @date created on 2023/1/26:0:16
 */
@WebServlet(name="userServlet",value = "/users")
public class UserServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
        /**设置下面这个以后，表单传过来的参数中文值就不是乱码了！*/
        request.setCharacterEncoding("UTF-8");
        /**常用方法*/
        // 获取请求时的完整路径（从http开始，到"?"前面结束）
        String url = request.getRequestURL().toString();
        System.out.println("获取请求时的完整路径:"+url);
        // 获取请求时的部分路径 （从项目的站点名开始，到"?"前面结束）
        String uri = request.getRequestURI();
        System.out.println("获取请求时的部分路径:"+uri);
        // 获取请求时的参数字符串（从"?"后面开始，到最后的字符串）
        String queryString = request.getQueryString();
        System.out.println("获取请求时的参数字符串:"+queryString);
        // 获取请求方式（Get or Post）
        String method = request.getMethod();
        System.out.println("获取请求方式:"+method);
        // 获取当前请求的协议版本 （HTTP/1.1）
        String protocol = request.getProtocol();
        System.out.println("获取当前请求的协议版本:"+protocol);
        // 获取项目的站点名称
        String contextPath = request.getContextPath();
        System.out.println("获取项目的站点名称:"+contextPath);
        // 获取servlet的名称
        String servletPath = request.getServletPath();
        System.out.println("获取servlet的名称:"+servletPath);
        System.out.println("users servlet!");
        response.getWriter().write("hello users servlet!");
    }
}

第4章 Tomcat生命周期组件

​ Lifecycle为Tomcat源码中组件的生命周期顶层接口，其位于org.apahce.catalina包下，地位堪比Spring源码中的BeanFactory，之所以比较喜欢Tomcat源码，也是源自于这个接口。

​ 其存在的意义在于使得Catalina容器能够更加方便的统一管理组件的生命周期行为，以提供一种标准的流程来启动和关闭组件，具体点就是组件的【初始化 ==>启动==>停止==>销毁】全靠这个接口来规范，每一种行为的触发都会牵扯到当前组件生命周期状态（LifecycleState）的转换，而每一种状态的改变又会触发相应的生命周期事件（LifecycleEvent）的”发布”，事件一旦发布，就会有相应的生命周期监听器（LifecycleListener）来处理这个事件，最典型的两个监听器就是org.apache.catalina.startup包下的HostConfig和ContextConfig。看到这里的小伙伴应该会很兴奋，因为这里说的不就是设计模式中的观察者模式吗，如果再带上一条那就还牵扯到状态模式，如果继续深入下去，你会发现凡是源码中出现顶层接口的地方，其都会伴随着一个抽象类，而这个抽象类采用的设计模式绝逼就是模板方法！！！你看，学习源码的好处这不就来了吗。

​ 本章会围绕下面一张图展开并且粗略（因为后面还会基于每个组件进行细说，所以本章就只是一笔带过了）的说下Tomcat源码中生命周期相关的类都有哪些，以及他们各自的用途等。

Note：

不同于Spring中的Bean创建的生命周期，Tomcat中的生命周期实现的对象是组件，生命周期状态是固定的，但是组件却有很多个，比如StandardService、StandardServer、Container容器接口下的各个实现子类等，这些组件之间得配合着工作才能让Catalina容器稳定高效的提供HTTP/HTTPS服务。而Spring中生命周期的实现特指的是Bean对象的创建过程，比如BeanDefinition的扫描、BFPP的调用、基于反射为Bean分配内存空间、populateBean填充bean属性、invokeAwareMethods调用各种织入类型的SetXXX方法、BPP前后置增强器的调用、Bean的destroy等等。
但二者又有一个很明显的共同点，那就是不管是Tomact组件的生命周期管理也好，还是Spring·Bean生命周期的管理，其都需要一个容器去承载这些管理的工作，Tomcat中毋庸置疑的是Catalina容器，而Spring中是IOC容器，所谓的IOC容器其实就是更高级一点的BeanFactory的实现类而已。
当然不仅仅是Tomcat和Spring有容器，像JBoss也有容器，那就是EJB容器，你只要记住，容器相当于管理者就行了，干活的是组件、是Bean、是SessionBean等。

4.1 Lifecycle

4.1.1 方法

  • addLifecycleListener : 添加生命周期监听器，即每一个实现了Lifecycle接口的生命周期组件内部都维护了一个监听器列表，监听器的作用就是回调，而回调执行的时机大多数都是在业务代码执行的前后。
  • findLifecycleListeners：返回监听器列表，即将List转换成Array返出去。
  • removeLifecycleListeners：移除监听器列表中指定的监听器对象。
  • init：组件初始化方法，会触发init阶段的两个事件（初始化前/后）的发布。
  • start：组件启动方法，会触发start阶段的三个事件（启动前/中/后）的发布。
  • stop：组件停止方法，会触发stop阶段的三个事件（停止前/中/后）的发布。
  • destroy：组件销毁方法，会触发destroy阶段的两个事件（销毁前/后）的发布。
  • getState： 获取组件状态对象。
  • getStateName：获取组件状态名称。

4.1.2 静态常量成员

init阶段包含的事件有：BEFORE_INIT_EVENT（初始化前事件）、AFTER_INIT_EVENT（初始化后事件）。

start阶段包含的事件有： BEFORE_START_EVENT（启动前事件）、START（启动中事件）、 AFTER_START_EVENT（启动后事件）。

stop阶段包含的事件有：BEFORE_STOP_EVENT（停止前事件） STOP_EVENT（停止中事件） 、AFTER_STOP_EVENT（停止后事件）

destroy阶段包含的事件有：BEFORE_DESTROY_EVENT（销毁前事件）、 AFTER_DESTROY_EVENT（销毁后事件）

另外，除了上面我们比较熟悉的事件以外，Tomcat作者还在Lifecycle接口中增加了另外两个类型的事件，它们分别是periodic和configure：

periodic类型仅有一个事件，那就是PERIODIC_EVENT，这个事件主要用于在容器组件（StandardHost）中进行发布，然后由定时线程周期性的执行任务，比如捕获这个事件的监听器HostConfig类中的check方法，就是定期去扫描webapps目录，检查目录下是否有应用修改，如果server.xml中配置了自动部署为true，那么就会重新部署当前的应用。

    /**国际化日志管理类*/

    /**
     * The list of registered LifecycleListeners for event notifications.
     * 这个是写时复制列表，搞这个是因为“
     * tomcat有很多组件，每个组件都有可能有自己的listener，监听器在组件生命周期中会触发，
     * 当然也有可能会添加新的监听器，这时候效率最高且安全的就是CopyOnWriteArrayList。
     *
     * 1.适用于数据量不大的场景，不适用于数据量大的场景。由于写操作的时候，需要拷贝数组，
     *   会消耗内存，如果原数组的内容比较多的情况下，可能导致young gc或者full gc
     * 2.适用于读多写少的场景，不适用于实时读的场景。
     *
     * CopyOnWriteArrayList能保证写操作的线程安全，也能保证数据的最终一致性，但是无法保证数据的实时一致性。
     * CopyOnWriteArrayList在写操作中，使用了ReentrantLock锁以保证线程安全，并替换原array属性；但是读
     * 的时候直接读取array，可能会发生在写操作替换array前后。这就会导致读到旧数据，引发不一致。
     *
     * 1、如果写操作未完成，那么直接读取原数组的数据；
     * 2、如果写操作完成，但是引用还未指向新数组，那么也是读取原数组数据；
     * 3、如果写操作完成，并且引用已经指向了新的数组，那么直接从新数组中读取数据。
     */
    private final List<LifecycleListener> lifecycleListeners = new CopyOnWriteArrayList<>();


    /**
     * The current state of the source component.
     * 默认资源组件的状态为：NEW，因为最开始组件肯定是处于刚刚创建但是还没有init的状态
     */
    private volatile LifecycleState state = LifecycleState.NEW;

4.3 LifecycleState

​ 生命周期状态枚举类，类的内部定义了很细的状态，每一个状态都对应一个生命周期事件名称，即状态变了，事件也会相应的进行切换，而不同的事件会触发生命周期组件相应的监听器的执行。


/**
 * The list of valid states for components that implement {@link Lifecycle}.
 * See {@link Lifecycle} for the state transition diagram.
 * 实现了Lifecycle接口的组件有效状态列表（定义的太TM的细了！！！）
 */
public enum LifecycleState {
    NEW(false, null), // 实例化
    INITIALIZING(false, Lifecycle.BEFORE_INIT_EVENT), // 初始化前
    INITIALIZED(false, Lifecycle.AFTER_INIT_EVENT),// 初始化后
    STARTING_PREP(false, Lifecycle.BEFORE_START_EVENT),// 启动前
    STARTING(true, Lifecycle.START_EVENT),// 启动中
    STARTED(true, Lifecycle.AFTER_START_EVENT),// 启动后
    STOPPING_PREP(true, Lifecycle.BEFORE_STOP_EVENT),// 停止前
    STOPPING(false, Lifecycle.STOP_EVENT),// 停止中
    STOPPED(false, Lifecycle.AFTER_STOP_EVENT),// 已停止
    DESTROYING(false, Lifecycle.BEFORE_DESTROY_EVENT),// 销毁中
    DESTROYED(false, Lifecycle.AFTER_DESTROY_EVENT),// 已销毁
    FAILED(false, null);// 失败

    private final boolean available;
    private final String lifecycleEvent;
    
     private LifecycleState(boolean available, String lifecycleEvent) {
        this.available = available;
        this.lifecycleEvent = lifecycleEvent;
    }

4.6.1.1 什么是MBean

​ MBean：是Managed Bean的简称，可以翻译为“管理构件”。一个MBean是一个被管理的Java对象，有点类似于JavaBean。一个设备、一个应用或者任何资源都可以被表示为MBean，MBean会暴露一个接口，接口中提供可供读写的属性以及可以被调用的方法。通常一个MBean定义的接口名称需要以MBean结尾， 例如： HelloMBean, 格式为XXXMBean，这个是规范，必须得遵守，如果不遵守，在编写JMX案例的时候，注册MBean的时候会报如下的错误：

MBean class com.xxxx.jmx.Hello does not implement DynamicMBean

​ JMX规范定义了五种类型的MBean，分别是：①Standard MBean ②Dynamic MBean ③Open MBean ④Model MBean ⑤MXBean

​ 其中我们按规范定义的HelloMBean属于标准MBean，Tomcat中的BaseModelMBean属于动态MBean，其实现了DynamicMBean接口。

​ 总结：只有通过特定实现的Java才能被管理，这种特定实现的机制就是MBean。

4.6.1.2 什么是MBeanServer

​ MBeanServer是一个包含所有已注册MBean的仓库，直译就是管理MBean的服务，它是JMX代理层的核心。JMX1.0规范提供一个接口叫 javax.management.MBeanServer。所有被管理的MBean的操作都可以通过MBeanServer来执行。使用MBeanServer实例，你能够管理所有的MBean。

​ 常规操作是我们创建的Java对象只能由我们开发人员自行来管理（JVM管理的层面不在我们讨论的范畴），比如属性设置/查看，方法调用等都是需要我们在程序中进行事先编码的，这种方式很不灵活，但是交给MBeanServer后，就不一样了。这就好比有钱人家的娃，自己不带，委托给专业的保姆带，按理说不安全（因为是基于C端工具就可以连接操作Java应用程序，如果想保证安全，还要做很多工作，比如遵循JMX安全规范，对远程客户机进行认证，限制MBean信息的访问和操作执行者，暴力点就是不对外开放JMX端口号就行了），但奈何人家照顾娃经验丰富专业有章法啊。

​ 所以说，如果你想监控和管理你开发的Java应用程序，那就像Tomcat一样开启JMX功能即可，如果没有这种需求的话，你甚至可以不用知道JMX是什么。

6.3 JVM类加载工作流程

6.3.1 类的生命周期

​ 类从加载至JVM内存中，到最终卸载，一共经历了7个生命周期阶段，这七个阶段分别是：

加载--》验证--》准备--》解析--》初始化--》使用--》卸载

6.3.2 类加载具体流程

（1）加载：JVM将磁盘上的.class文件以二进制的形式（字节码）读取（写入）到内存。
（2）验证：确保类符合JVM规范和安全，比如字节码开头的几个字节必须是：CAFE BABY。
（3）准备：就是为类的静态变量分配空间，并设置默认值，比如static int a = 1，
在堆空间分配，并设置默认值0；而静态常量会被直接赋值，比如 static final int b=1，b会在准备阶段直接赋值为1
（4）解析：虚拟机将常量池的符号引用转变成直接引用。
注意：因为类中可能会有其他类的引用，加载到内存中时为了省空间，直接用符号代替，在这一步中，会将这些符号替换成真正的地址引用。
（5）初始化：这是类加载机制的最后一步，在这个阶段，java程序代码才真正开始执行；
在准备阶段，会给类的静态变量赋初始值，但是在类的初始化阶段，会赋予静态变量真正的值了；比如static int a = 1；在准备阶段a=0，但是在初始化阶段，a=1；同时会执行静态代码块，完成初始化工作；（分clinit 和 init,如果有静态变量、方法、代码块，就会进行clinit）；这里要干两件事情：
（1）如果存在父类，先初始化父类
（2）如果类中存在初始化语句，则依次执行他们，比如static{} 代码块

​ 总结：类加载的最终产物是java.lang.Class对象，它是我们访问方法区中的类型数据的外部接口，我们需要通过这个对象来访问类的字段、方法、运行时常量池等。在JDK1.8中，类加载的产物即java.lang.Class对象是存放在堆上的，而且类的静态变量是存在于Class对象中的，即也是存在于堆上的；而类的元数据，如：类的方法代码、方法名、变量名、访问权限等都是存放在元空间上的

6.4 父委派模式是什么

​ 英文：parent-delegation model，大部分人或文章直译为双亲委派模式，双亲这个词用在这里不太合适，我们知道对于类加载器来说，它的父层级仅有一个，就算是父的父的父，那也是多亲，而不是双。如果觉得父委派模式还是很拗口难以接受，那最好的办法就是直接称作委派（委托）模式好了，委派放在JVM类加载器场景中是很好理解的，比如：

​ 如果一个类加载器收到了类加载的请求，首先他并不会自己加载，而是将这个请求委托给自己的父加载器去完成加载，每一层的类加载器都会把请求委派给自己的父加载器，直到所有的类加载请求传递给顶层的启动类加载器，如果父加载器无法完成加载请求，则子加载器才会尝试去加载，如果连子ClassLoader都无法完成类加载请求的话，那就会抛出ClassNotFound的异常。

​ 至于为什么说父委派模式中的类加载走的是组合而不是继承，我们可以通过JDK源码考证：

package java.lang;
public abstract class ClassLoader {
     protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException
    {
        /**
          * 1. 首先，获取类加载的锁，锁对象有两种
          * 第一种：类加载自身，这种不支持并发属于串行加载class，效率低
          * 第二种：所加载class的完全限定名字符串对象，这种不锁类加载器，使得类加载器可以并行加载
          */
        synchronized (getClassLoadingLock(name)) {
            // 2. 获取锁权的第一件事情，就是看一下class是否已经加载过了，如果加载过了直接返回
            Class<?> c = findLoadedClass(name);
            // 2.1 如果类没有并加载，白话就是第一次加载
            if (c == null) {
                long t0 = System.nanoTime();
                try {
                    /**
                      * 2.1.1 第一次加载的机会委托给父加载器去加载，如果父不为空就父去加载
                      * 这里有个递归调用，也就是存在多层父级的话，就会层层往上找，直到找到
                      * 最顶层的父加载器，然后调用findBootstrapClassOrNull去加载当前的class
                      */
                    if (parent != null) {
                        c = parent.loadClass(name, false);
                    } else {
                        // 2.1.2 这个方法是native方法，如果父加载器加载不了，那c=null
                        c = findBootstrapClassOrNull(name);
                    }
                } catch (ClassNotFoundException e) {
                    // ClassNotFoundException thrown if class not found
                    // from the non-null parent class loader
                }

                if (c == null) {
                    long t1 = System.nanoTime();
                    // 3. 如果父类加载器找不到class无法加载，那就交给当前的类加载器去加载
                    // 如果当前类加载器也无法加载，那就抛ClassNotFoundException异常了！
                    c = findClass(name);
                    // this is the defining class loader; record the stats
                    sun.misc.PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                    sun.misc.PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                    sun.misc.PerfCounter.getFindClasses().increment();
                }
            }
            // 4. 如果resolve为true，则加载class的同时对其进行解析操作
            if (resolve) {
                resolveClass(c);
            }
            return c;
        }
    }
}

6.5 为什么要打破父委派模式

​ 这里我们不谈JNDI，也不谈JDBC规范，因为他们本质上都是SPI机制的一种实现，他们都打破了父委派模式，而且是不得不打破。这里我们来说说Tomcat，确切的说应该是所有实现了Servlet规范的Web容器。他们为什么要打破呢？

​ 首先应用服务器（jetty，tomcat等）通常会自行创建类加载器以实现更灵活的控制，这一方面是对规范的实现（Servlet规范要求每个Web应用都有一个独立的类加载器实例），另一方面也有架构层面的考虑：

（1）隔离性：Web应用类库相互隔离，避免依赖库或者应用包相互影响。

如果所有的应用都共用一个类加载器实例去加载所有应用的class的话，那么必然会出现这样的问题：A引用依赖了mybatis2，B应用依赖了mybaits3，由于myabtis2和3之间差异过大，导致只有一个应用类加载器的情况下，myabtis相关的类只会被加载一次，因为类的完全限定名是固定的，但是类中的方法可不是，这就有可能导致内存中的B依赖的mybatis相关类的版本是2，这必然会导致B应用在被调用时触发NoSuchMethodException的异常。

（2）灵活性：既然Web应用之间的类加载器相互独立，那么我们就能只针对一个Web应用进行重新部署，此时该Web应用的类加载器将会被重新创建，而且不会影响其他Web应用。

如果所有的应用都共用一个类加载的话，一个应用热更新的话，可能会对另外一个应用的class产生不可预测的影响。

（3）性能：由于每个Web应用都有一个类加载器，因此Web应用在加载类时，不会搜索其他Web应用包含的Jar包，性能自然高于应用服务器只有一个类加载器的情况。如果多个应用之间有共享的依赖，那我们可以将这些共享的jar提取到指定的目录下，然后交给应用类加载器的父加载器去加载，这样就能减少共性class 的加载次数，提高Tomcat应用服务器的启动速度！如果这些jar万年不动的话，我们还可以构建应该程序包的时候将这些jar排除，从另一方面来说，也提高了应用的build速度！

那么这就有一个问题了，如果有N个应用，都是Spring项目，他们依赖的Spring模块包版本都一样，如果我们将其都提取给父类加载器去加载的话，会不会有问题呢？
1. 首先，先撇开Spring源码，我们来思考一下会有什么问题。Spring是ioc容器，是要管理应用程序的bean的，但是我们把Spring的相关依赖提取到了父加载器中，我们知道，传统父委派模式中，父加载器是无法加载子加载器作用域下的class的，那么我们父加载器在加载Spring依赖的时候，同时也会去加载应用程序的beanCLass，这时候矛盾就出现了，我父加载器做不到啊？？？

7.1.1 LogFactory

​ 位于包org.apache.juli.logging下，日志工厂类，即是单例又是工厂模式。通过getFactory()方法获取到唯一的日志工厂实例，通过工厂调用getLog(Class<?> clazz)获取到类对应的具体日志实现类，默认是DirectJDKLog类型，其是对JUL（java.util.logging包下面的Logger）的封装，也就是Tomcat自己实现了一个日志类来代替JDK默认提供的日志组件。

@ServiceConsumer(value=org.apache.juli.logging.Log.class)
public class LogFactory {
    /**单例：饿汉模式*/
    private static final LogFactory singleton = new LogFactory();
    private LogFactory() {
        FileSystems.getDefault();
        /**这个地方我们可以扩展，自定义我们自己的日志组件，然后替换Tomcat默认的*/
        ServiceLoader<Log> logLoader = ServiceLoader.load(Log.class);
        Constructor<? extends Log> m=null;
        for (Log log: logLoader) {
            Class<? extends Log> c=log.getClass();
            try {
                m=c.getConstructor(String.class);
                break;
            }
            catch (NoSuchMethodException | SecurityException e) {
                throw new Error(e);
            }
        }
        /** 如何没有定义 Log 的实现类，discoveredLogConstructor 为 null*/
        discoveredLogConstructor=m;
    }

    public static LogFactory getFactory() throws LogConfigurationException {
        return singleton;
    }

    public static Log getLog(Class<?> clazz)
        throws LogConfigurationException {
        return getFactory().getInstance(clazz);
    }
    
    public static Log getLog(String name)
        throws LogConfigurationException {
        return getFactory().getInstance(name);
    }

    public Log getInstance(String name) throws LogConfigurationException {
        if (discoveredLogConstructor == null) {
            /**
             * 如果 discoveredLogConstructor 为 null，也就没有定义 Log 类，默认用DirectJDKLog
             * DirectJDKLog 是对java.util.logging.Logger的封装，只不过加了默认格式化实现
             */
            return DirectJDKLog.getInstance(name);
        }
        try {
            return discoveredLogConstructor.newInstance(name);
        } catch (ReflectiveOperationException | IllegalArgumentException e) {
            throw new LogConfigurationException(e);
        }
    }
}

7.1.3 StringManager

​ 位于org.apache.tomcat.util.res包下面，主要负责管理Tomcat服务的错误消息，消息对于使用者来说非常重要，比如最常见的，Tomcat无法启动时因为端口已绑定，通过这个错误消息我们就可以排查是否是系统存在和Tomcat连接器一样的端口占用着等等。当然消息是支持国际化的，做过前后端国际化的小伙伴应该知道，其维护的原理就是一个key一个value，key是固定的，重点就在value要做国际化，key固定，value随着地区要做翻译，key多个，那毕然会有很多个消息文件，每一个文件对应一个地区的语言翻译；而StringManager干的就是这种事情，通过key拿到value，然后再通过Tomcat的日志类输出错误消息到Console或者File。

​ Tomcat中的类有几百个，不可能每个类都实例化一个StringManager，必然是要用一个实例的，那问题又来了，难道几百个类每个类都要单独搞个错误消息的国际化文件吗？如果这样维护太不方便了，Tomcat做法是按包分类，拿catalina.startup包举例，其下的所有类共用一个Resource Bundle，所谓的Bundle就是一捆，多个错误消息国际化资源文件放在一起称作一个bundle，比如下面的：

​ StringManager类代码精简如下：

public class StringManager {

    private static int LOCALE_CACHE_SIZE = 10;
    private final ResourceBundle bundle;
    private final Locale locale;

    private StringManager(String packageName, Locale locale) {
        String bundleName = packageName + ".LocalStrings";
        ResourceBundle bnd = null;
        try {
            if (locale.getLanguage().equals(Locale.ENGLISH.getLanguage())) {
                locale = Locale.ROOT;
            }
            bnd = ResourceBundle.getBundle(bundleName, locale);
        } catch (MissingResourceException ex) {
            ClassLoader cl = Thread.currentThread().getContextClassLoader();
            if (cl != null) {
                try {
                    bnd = ResourceBundle.getBundle(bundleName, locale, cl);
                } catch (MissingResourceException ex2) {
                }
            }
        }
        bundle = bnd;
        if (bundle != null) {
            Locale bundleLocale = bundle.getLocale();
            if (bundleLocale.equals(Locale.ROOT)) {
                this.locale = Locale.ENGLISH;
            } else {
                this.locale = bundleLocale;
            }
        } else {
            this.locale = null;
        }
    }

    public String getString(String key) {
        if (key == null){
            String msg = "key may not have a null value";
            throw new IllegalArgumentException(msg);
        }

        String str = null;
        try {
            if (bundle != null) {
                str = bundle.getString(key);
            }
        } catch (MissingResourceException mre) {
            str = null;
        }
        return str;
    }


    public String getString(final String key, final Object... args) {
        String value = getString(key);
        if (value == null) {
            value = key;
        }
        try{
            /** 修改编码，防止tomcat启动时控制台日志中文乱码*/
            value = new String(value.getBytes("ISO-8859-1"),"UTF-8");
        }catch (Exception e){
        }
        MessageFormat mf = new MessageFormat(value);
        mf.setLocale(locale);
        return mf.format(args, new StringBuffer(), null).toString();
    }


    private static final Map<String, Map<Locale,StringManager>> managers = new Hashtable<>();

    public static final StringManager getManager(Class<?> clazz) {
        return getManager(clazz.getPackage().getName());
    }

    public static final StringManager getManager(String packageName) {
        return getManager(packageName, Locale.getDefault());
    }

    public static final synchronized StringManager getManager(String packageName, Locale locale){
        Map<Locale,StringManager> map = managers.get(packageName);
        if (map == null) {
            map = new LinkedHashMap<Locale,StringManager>(LOCALE_CACHE_SIZE, 1, true) {
                private static final long serialVersionUID = 1L;
                @Override
                protected boolean removeEldestEntry(
                        Map.Entry<Locale,StringManager> eldest) {
                    if (size() > (LOCALE_CACHE_SIZE - 1)) {
                        return true;
                    }
                    return false;
                }
            };
            managers.put(packageName, map);
        }
        StringManager mgr = map.get(locale);
        if (mgr == null) {
            mgr = new StringManager(packageName, locale);
            map.put(locale, mgr);
        }
        return mgr;
    }

    public static StringManager getManager(String packageName,
            Enumeration<Locale> requestedLocales) {
        while (requestedLocales.hasMoreElements()) {
            Locale locale = requestedLocales.nextElement();
            StringManager result = getManager(packageName, locale);
            if (result.getLocale().equals(locale)) {
                return result;
            }
        }
        return getManager(packageName);
    }
}

         * (以Common类加载器为父类，适用于加载Tomcat自身依赖的Jar的类加载器，其路径由server.loader指定，默认为空，此时tomcat使用Common类加载器加载自身所需的jar。)
         * shared.loader= , 这个shared.loader没有值，tomcat作者说了，如果这个留空，那shared的类加载器就是commonLoader
         * 以Common类加载器为父类，是所有Web应用的父类加载器，其路径由shared.loader指定，默认为空，此时tomcat使用Common类加载器作为Web应用的父加载器。
         *
         * Web应用：以Shared类加载器为父类，加载/WEB-INF/classes目录下的未压缩的Class和资源文件以及/WEB-INF/lib目录下的jar包，
         *         该类加载器只对当前Web应用可见，对其他Web应用均不可见。
         * 默认情况下，Common、Catalina、Shared类加载器是同一个，但可以配置3个不同的类加载器，使他们各司其职
         */
        String value = CatalinaProperties.getProperty(name + ".loader");
        /**如果找不到name对应的loader配置选项的话，就返回父类加载器*/
        if ((value == null) || (value.equals(""))) {
            return parent;
        }

        /**value值是一个带变量表达式的字符串，如 "xxx/${catalina.base}/lib",replace就是要替换变量换成具体的值 */
        value = replace(value);

        /**Repository是ClassLoaderFactory 中的一个静态内部类, 有2个属性, location, type, 表示某个位置的某种类型的文件*/
        List<Repository> repositories = new ArrayList<>();

        /**
         * 默认情况下，如果我们不配置catalina.properties中的common.loader标签的值的话，这里取出的数组就4个
         * 分别是：
         * 1、${catalina.base}/lib
         * 2、${catalina.base}/lib/*.jar
         * 3、${catalina.home}/lib
         * 4、${catalina.home}/lib/*.jar
         *
         * bin (运行脚本）
         * conf (配置文件）
         * lib (核心库文件）
         * logs (日志目录)
         * temp (临时目录)
         * webapps (自动装载的应用程序的目录）
         * work (JVM临时文件目录[java.io.tmpdir])
         * 其中只有bin和lib被多个tomcat示例共用，其它目录conf、logs、temp、webapps和work 每个Tomcat实例都拥有独自一份
         * 所以：
         * catalina.home(安装目录)：指向公用信息的位置，就是bin和lib的父目录。
         * catalina.base(工作目录)：指向每个Tomcat目录私有信息的位置，就是conf、logs、temp、webapps和work的父目录。
         * 在只有一个tomcat实例的情况下，home和base是一样的
         * */
        String[] repositoryPaths = getPaths(value);

        for (String repository : repositoryPaths) {
            // Check for a JAR URL repository
            try {
                @SuppressWarnings("unused")
                /**
                 * 一上来先暴力检查是否是一个url类型的仓库，如果不是，
                 * tomcat原来异常是忽略不处理，后面会再判断是不是glob、jar或者dir类型的
                 * 类型是一个枚举类，有：
                 * DIR, -- 目录格式的，如 file://xxxxxx
                 * GLOB, -- N个jar，*.jar结尾的都算
                 * JAR, -- 单个jar
                 * URL -- 网络地址形式的，如http://xxxxxx
                 */
                URL url = new URL(repository);
                repositories.add(new Repository(repository, RepositoryType.URL));
                continue;
            } catch (MalformedURLException e) {
                // Ignore
                System.out.printf("%s - 不是URL repository \n",repository);
            }
            // Local repository , 既然不是网络的，那一定是本地文件系统可以访问到的
            if (repository.endsWith("*.jar")) {
                repository = repository.substring
                    (0, repository.length() - "*.jar".length());
                repositories.add(new Repository(repository, RepositoryType.GLOB));
            } else if (repository.endsWith(".jar")) {
                repositories.add(new Repository(repository, RepositoryType.JAR));
            } else {
                repositories.add(new Repository(repository, RepositoryType.DIR));
            }
        }
        /**这个地方调用了ClassLoaderFactory的创建类加载方法*/
        return ClassLoaderFactory.createClassLoader(repositories, parent);
    }
}

public final class ApplicationFilterFactory {

    private ApplicationFilterFactory() {
        // Prevent instance creation. This is a utility class.
    }
    public static ApplicationFilterChain createFilterChain(ServletRequest request,
            Wrapper wrapper, Servlet servlet) {

        // If there is no servlet to execute, return null
        if (servlet == null) {
            return null;
        }

        // Create and initialize a filter chain object
        ApplicationFilterChain filterChain = null;
        if (request instanceof Request) {
            Request req = (Request) request;
            if (Globals.IS_SECURITY_ENABLED) {
                // Security: Do not recycle
                filterChain = new ApplicationFilterChain();
            } else {
                filterChain = (ApplicationFilterChain) req.getFilterChain();
                if (filterChain == null) {
                    filterChain = new ApplicationFilterChain();
                    req.setFilterChain(filterChain);
                }
            }
        } else {
            // Request dispatcher in use
            filterChain = new ApplicationFilterChain();
        }

        filterChain.setServlet(servlet);
        filterChain.setServletSupportsAsync(wrapper.isAsyncSupported());

        // Acquire the filter mappings for this Context
        StandardContext context = (StandardContext) wrapper.getParent();
        FilterMap filterMaps[] = context.findFilterMaps();

        // If there are no filter mappings, we are done
        if ((filterMaps == null) || (filterMaps.length == 0)) {
            return filterChain;
        }

        // Acquire the information we will need to match filter mappings
        DispatcherType dispatcher =
                (DispatcherType) request.getAttribute(Globals.DISPATCHER_TYPE_ATTR);

        String requestPath = null;
        Object attribute = request.getAttribute(Globals.DISPATCHER_REQUEST_PATH_ATTR);
        if (attribute != null){
            requestPath = attribute.toString();
        }

        String servletName = wrapper.getName();

        // Add the relevant path-mapped filters to this filter chain
        for (FilterMap filterMap : filterMaps) {
            if (!matchDispatcher(filterMap, dispatcher)) {
                continue;
            }
            if (!matchFiltersURL(filterMap, requestPath)) {
                continue;
            }
            ApplicationFilterConfig filterConfig = (ApplicationFilterConfig)
                    context.findFilterConfig(filterMap.getFilterName());
            if (filterConfig == null) {
                // FIXME - log configuration problem
                continue;
            }
            filterChain.addFilter(filterConfig);
        }

        // Add filters that match on servlet name second
        for (FilterMap filterMap : filterMaps) {
            if (!matchDispatcher(filterMap, dispatcher)) {
                continue;
            }
            if (!matchFiltersServlet(filterMap, servletName)) {
                continue;
            }
            ApplicationFilterConfig filterConfig = (ApplicationFilterConfig)
                    context.findFilterConfig(filterMap.getFilterName());
            if (filterConfig == null) {
                // FIXME - log configuration problem
                continue;
            }
            filterChain.addFilter(filterConfig);
        }
        // Return the completed filter chain
        return filterChain;
    }
}

7.2.4 ProtocolHandler

​ 位于org.apache.coyote包下，协议处理器。其是一个特殊的工厂模式，因为我们知道工厂模式涉及到的都是类，而我们的这个ProtocolHandler却是接口，我们知道Tomcat9源码是基于JDK8的，而JDK8中对接口新增了特性，分别是：支持静态方法和默认方法，而恰恰利用了这个特性，在ProtocolHandler接口中我们可以看到有一个静态方法static ProtocolHandler create(String protocol, boolean apr)，好巧不巧，这个方法非常符合工厂方法的设计模式，如果你去看视频或搜索，不会有人告诉你这个接口也是建立在工厂设计模式之上的。

public interface ProtocolHandler {
public static ProtocolHandler create(String protocol, boolean apr)
            throws ClassNotFoundException, InstantiationException, IllegalAccessException,
            IllegalArgumentException, InvocationTargetException, NoSuchMethodException, SecurityException {
        /**基于协议类型创建协议处理器实例*/
        if (protocol == null || "HTTP/1.1".equals(protocol)
                || (!apr && org.apache.coyote.http11.Http11NioProtocol.class.getName().equals(protocol))
                || (apr && org.apache.coyote.http11.Http11AprProtocol.class.getName().equals(protocol))) {
            if (apr) {
                /**
                 * 如果apr为true，就使用apr协议
                 * apr=Apache Portable Runtime的简称，是Apache HTTP Server2.x的核心。
                 * 是一个高可移植库，APR有很多用途，包括访问高级IO功能(例如sendfile，epoll和OpenSSL)，
                 * OS级别功能(随机数生成，系统状态等等)，本地进程管理(共享内存，NT管道和UNIXsockets)。
                 * 这些功能可以使Tomcat作为一个通常的前台WEB服务器，能更好地和其它本地web技术集成，总体上让Java更有效率作为一个高性能web服务器平台而不是简单作为后台容器。
                 * */
                return new org.apache.coyote.http11.Http11AprProtocol();
            } else {
                /**默认是nio协议的处理器，当然tomcat还有一个nio2协议*/
                return new org.apache.coyote.http11.Http11NioProtocol();
            }
        } else if ("AJP/1.3".equals(protocol)
                || (!apr && org.apache.coyote.ajp.AjpNioProtocol.class.getName().equals(protocol))
                || (apr && org.apache.coyote.ajp.AjpAprProtocol.class.getName().equals(protocol))) {
            if (apr) {
                return new org.apache.coyote.ajp.AjpAprProtocol();
            } else {
                return new org.apache.coyote.ajp.AjpNioProtocol();
            }
        } else {
            // Instantiate protocol handler
            /**如果不是特定的协议而是协议处理器类完全限定名的话，就反射出对应的处理器对象*/
            Class<?> clazz = Class.forName(protocol);
            return (ProtocolHandler) clazz.getConstructor().newInstance();
        }
    }
}

7.4 模板方法

​ 模板方法（Tepmlate Pattern - 行为型）在各种框架中是最最最常见的设计模式之一，其意图是定义一个操作中的算法骨架，而将一些步骤延迟到子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。而在Tomcat源码中，模板方法是应用最多的一个设计模式之一。

7.4.1 LifecycleBase

​ 位于org.apache.catalina.util包下，所有生命周期组件的父类，类的类型为abstract，不仅实现了Lifecycle接口中定义的所有方法，同时还为子组件的生命周期增加了共同的算法步骤，且这些算法步骤（抽象方法）会延迟到子组件中去实现，比如initInternal、startInternal、stopInternal、destroyInternal等。（严格意义上的模板方法，为了防止恶意操作，抽象类中的定义算法步骤的方法上要加上final关键字，Tomcat源码就是这么干的）。

​ 精简代码后的代码如下：

public abstract class LifecycleBase implements Lifecycle {
    private final List<LifecycleListener> lifecycleListeners = new CopyOnWriteArrayList<>();
    private volatile LifecycleState state = LifecycleState.NEW;
    @Override
    public final synchronized void init() throws LifecycleException {
        /**
         * 下面的意思是：你组件初始化的时候必须是处于new的状态，为什么呢? 因为状态机中你第一次初始化时
         * 不处于new难不成还处于start或stop状态？ 那不就乱套了，这里当然会抛异常了！
         * */
        if (!state.equals(LifecycleState.NEW)) {
            invalidTransition(Lifecycle.BEFORE_INIT_EVENT);
        }
        try {
            /**
             *  发布 Lifecycle.BEFORE_INIT_EVENT 事件，
             *  如果当前组件注册的有listeners就回调反之不处理
             *  事实上仅有Server组件的lifecycleListeners不为空
             * */
            setStateInternal(LifecycleState.INITIALIZING, null, false);
            /**具体子类来实现，（模板方法）*/
            initInternal();
            /** 发布 Lifecycle.AFTER_INIT_EVENT 事件，如果当前组件注册的有listeners就回调反之不处理*/
            setStateInternal(LifecycleState.INITIALIZED, null, false);
        } catch (Throwable t) {
            handleSubClassException(t, "lifecycleBase.initFail", toString());
        }
    }
    protected abstract void initInternal() throws LifecycleException;
    
    @Override
    public final synchronized void start() throws LifecycleException {
        if (LifecycleState.STARTING_PREP.equals(state) || LifecycleState.STARTING.equals(state) ||
                LifecycleState.STARTED.equals(state)) {
            ... ... ...
            return;
        }
        
        if (state.equals(LifecycleState.NEW)) {
            init();
        } else if (state.equals(LifecycleState.FAILED)) {
            stop();
        } else if (!state.equals(LifecycleState.INITIALIZED) &&
                !state.equals(LifecycleState.STOPPED)) {
            invalidTransition(Lifecycle.BEFORE_START_EVENT);
        }
        try {
            setStateInternal(LifecycleState.STARTING_PREP, null, false);
            startInternal();
            ... ... ...
        } catch (Throwable t) {
            handleSubClassException(t, "lifecycleBase.startFail", toString());
        }
    }
    protected abstract void startInternal() throws LifecycleException;
    /**其他不再放*/

7.5.1 RecycledProcessors

​ 属于位于org.apache.coyote包下的类AbstractProtocol中的内部类，这个类功能很简单，其继承了Tomcat内部自定义的SynchronizedStack类，SynchronizedStack类的目的就是保证多线程并发下添加的到stack里的T类型的对象安全，而RecycledProcessors则扩展了其父类的方法，如push、pop和clear，更重要的是它持有一个ConnectionHandler对象，目的就是为了拿到handler对应的Protocol对象的processors缓存大小，该大小将决定stack栈是否可以继续添加T类型的对象。

protected static class RecycledProcessors extends SynchronizedStack<Processor> {

        private final transient ConnectionHandler<?> handler;
        protected final AtomicInteger size = new AtomicInteger(0);

        public RecycledProcessors(ConnectionHandler<?> handler) {
            this.handler = handler;
        }

        @SuppressWarnings("sync-override") // Size may exceed cache size a bit
        @Override
        public boolean push(Processor processor) {
            /**从请求连接处理器对象关联的应用层协议处理器对象上拿到processor缓存值*/
            int cacheSize = handler.getProtocol().getProcessorCache();
            /**如果cahcheSize为-1，则不受限直接push，如果大于等于cacheSize则无法push*/
            boolean offer = cacheSize == -1 ? true : size.get() < cacheSize;
            //avoid over growing our cache or add after we have stopped
            boolean result = false;
            if (offer) {
                result = super.push(processor);
                if (result) {
                    /**原子级+1*/
                    size.incrementAndGet();
                }
            }
            /**如果push失败，则要卸载*/
            if (!result) {
                handler.unregister(processor);
            }
            return result;
        }

        @SuppressWarnings("sync-override") // OK if size is too big briefly
        @Override
        public Processor pop() {
            Processor result = super.pop();
            if (result != null) {
                size.decrementAndGet();
            }
            return result;
        }

        @Override
        public synchronized void clear() {
            Processor next = pop();
            /**遍历卸载*/
            while (next != null) {
                handler.unregister(next);
                next = pop();
            }
            super.clear();
            size.set(0);
        }
    }

public final class ExpressionBuilder implements NodeVisitor {
	  public ValueExpression createValueExpression(Class<?> expectedType)
            throws ELException {
        /**EL解析工作在这个build方法里*/
        Node n = this.build();
        return new ValueExpressionImpl(this.expression, n, this.fnMapper,
                this.varMapper, expectedType);
    }
    private Node build() throws ELException {
        /**重点看这个，这个是干活的*/
        Node n = createNodeInternal(this.expression);
        /**这个是node计算前的准备工作，主要通过访问者模式对node进行类型判断然后做一些func参数的校验*/
        this.prepare(n);
        if (n instanceof AstDeferredExpression
                || n instanceof AstDynamicExpression) {
            n = n.jjtGetChild(0);
        }
        return n;
    }
     private static final Node createNodeInternal(String expr)
            throws ELException {
        if (expr == null) {
            throw new ELException(MessageFactory.get("error.null"));
        }

        /**先从缓存中取*/
        Node n = expressionCache.get(expr);
        /**如果表达式缓存中没有就从解析缓存栈里取*/
        if (n == null) {
            ELParser parser = parserCache.pop();
            try {
                if (parser == null) {
                    /**如果都没有，那就是第一次调用，new一个ELParser(大名鼎鼎的El表达式解析器)*/
                    parser = new ELParser(new StringReader(expr));
                } else {
                    parser.ReInit(new StringReader(expr));
                }

                /**解析*/
                n = parser.CompositeExpression();

                // validate composite expression
                int numChildren = n.jjtGetNumChildren();
                if (numChildren == 1) {
                    n = n.jjtGetChild(0);
                } else {
                    Class<?> type = null;
                    Node child = null;
                    for (int i = 0; i < numChildren; i++) {
                        child = n.jjtGetChild(i);
                        if (child instanceof AstLiteralExpression) {
                            continue;
                        }
                        if (type == null) {
                            type = child.getClass();
                        } else {
                            if (!type.equals(child.getClass())) {
                                throw new ELException(MessageFactory.get(
                                        "error.mixed", expr));
                            }
                        }
                    }
                }

                if (n instanceof AstDeferredExpression
                        || n instanceof AstDynamicExpression) {
                    n = n.jjtGetChild(0);
                }
                expressionCache.put(expr, n);
            } catch (Exception e) {
                throw new ELException(
                        MessageFactory.get("error.parseFail", expr), e);
            } finally {
                if (parser != null) {
                    /**每个el表达式解析后，对解析后的node进行缓存*/
                    parserCache.push(parser);
                }
            }
        }
        return n;
    }
    
    private void prepare(Node node) throws ELException {
        try {
            /**调用node的accept方法，将对象本身交给visitor对象进行后续操作，数据和操作进行解耦*/
            node.accept(this);
        } catch (Exception e) {
            if (e instanceof ELException) {
                throw (ELException) e;
            } else {
                throw (new ELException(e));
            }
        }
        if (this.fnMapper instanceof FunctionMapperFactory) {
            this.fnMapper = ((FunctionMapperFactory) this.fnMapper).create();
        }
        if (this.varMapper instanceof VariableMapperFactory) {
            this.varMapper = ((VariableMapperFactory) this.varMapper).create();
        }
    }
    
    @Override
    public void visit(Node node) throws ELException {
        /**这个地方需要配合自定义的tld文件进行调试，定义函数类，然后jsp页面进行关联*/
        if (node instanceof AstFunction) {
            AstFunction funcNode = (AstFunction) node;
            Method m = null;
            if (this.fnMapper != null) {
                m = fnMapper.resolveFunction(funcNode.getPrefix(), funcNode
                        .getLocalName());
            }
            
            // References to variables that refer to lambda expressions will be
            // parsed as functions. This is handled at runtime but at this point
            // need to treat it as a variable rather than a function.
            if (m == null && this.varMapper != null &&
                    funcNode.getPrefix().length() == 0) {
                this.varMapper.resolveVariable(funcNode.getLocalName());
                return;
            }

            if (this.fnMapper == null) {
                throw new ELException(MessageFactory.get("error.fnMapper.null"));
            }

            if (m == null) {
                throw new ELException(MessageFactory.get(
                        "error.fnMapper.method", funcNode.getOutputName()));
            }

            int methodParameterCount = m.getParameterTypes().length;
            // AstFunction->MethodParameters->Parameters()
            int inputParameterCount = node.jjtGetChild(0).jjtGetNumChildren();
            /**主要就是判断jsp中调用自定义tld 函数的参数是否跟原生函数的参数一致，不一致就抛异常*/
            if (m.isVarArgs() && inputParameterCount < methodParameterCount - 1 ||
                    !m.isVarArgs() && inputParameterCount != methodParameterCount) {
                throw new ELException(MessageFactory.get(
                        "error.fnMapper.paramcount", funcNode.getOutputName(),
                        "" + methodParameterCount, "" + node.jjtGetChild(0).jjtGetNumChildren()));
            }
        } else if (node instanceof AstIdentifier && this.varMapper != null) {
            String variable = node.getImage();

            // simply capture it
            this.varMapper.resolveVariable(variable);
        }
    }
}

7.16 反应器模式

​ 反应器模式（Reactor Pattern）是是处理并发I/O常见的一种模式，用于同步I/O，其中心思想是将所有要处理的I/O事件注册到一个中心I/O多路复用器上，同时主线程阻塞在多路复用器上，一旦有I/O事件到来或是准备就绪，多路复用器将返回并将相应I/O事件分发到对应的处理器中。以下是一些适合应用 “Reactor” 反应器设计模式的场景：

  1. 网络编程：”Reactor” 模式特别适用于网络编程，如服务器应用程序。服务器需要同时处理多个客户端的连接和请求，”Reactor” 模式可以有效地管理这些并发请求，从而提高系统的性能和响应性。

  2. 多用户系统：当系统需要同时处理多个用户的请求时，”Reactor” 模式可以帮助管理并发请求，确保每个用户都能获得适当的响应。

  3. GUI 应用程序：图形用户界面 (GUI) 应用程序通常需要处理用户输入、鼠标事件、键盘事件等。”Reactor” 模式可以帮助管理这些不同类型的事件，使应用程序能够快速响应用户的操作。

  4. 实时数据处理：当需要实时处理传感器数据、日志流、交易数据等时，”Reactor” 模式可以帮助有效地处理这些连续产生的事件。

  5. 游戏开发：游戏通常需要处理多个玩家的输入、游戏事件、碰撞检测等。”Reactor” 模式可以帮助管理这些游戏事件，确保游戏保持流畅的运行。

  6. 消息队列系统：在消息队列系统中，消息的生成和处理通常是异步的。”Reactor” 模式可以协调消息的生成和消费，确保系统的可伸缩性和稳定性。

  7. 实时通信应用：即时通信应用（如聊天应用、实时协作工具等）需要处理多个用户之间的实时消息交换。”Reactor” 模式可以管理这些并发的消息传递。

  8. 异步任务处理：当系统需要执行耗时的异步任务时，如文件 I/O、数据库查询等，”Reactor” 模式可以协调这些异步任务的执行。

​ 总之，”Reactor” 反应器设计模式适用于需要管理大量并发事件，并且希望通过异步操作提高系统性能和响应性的情况。它将事件的处理与事件的发生和处理解耦，使系统更容易扩展和维护。

​ 常用的Reactor模式有：单Reactor单线程单，Reactor多线程模式 和 多Reactor（主从）多线程模式。

7.16.1 BIO 和 NIO的对比

同步：调用方发起请求，被调用方在处理完毕后，才将处理结果返回给调用方。

异步：调用方发起请求，被调用方在处理完毕后，不直接将处理结果返回给调用方，被调用方可以继续处理其它请求，而处理结果则通过事件或者回调等方式来返回给调用者。

总结：同步和异步强调的是系统内核数据应该以什么样的形式（操作）返回给应用层。是应用层一直询问系统内核数据是否准备好，还是由系统内核准备好数据后主动通知应用层来取？

阻塞：调用方调用结果返回之前，当前线程会被挂起。

非阻塞：非阻塞和阻塞的概念相对应，指在不能马上获得结果以前，当前线程不会阻塞住，而会继续向下执行。回调就是一种非阻塞的异步模式。并发线程经过回调，能够将结果返回给发起线程。除了回调，还有其余的非阻塞异步模式，好比消息通信、信号量等等。

总结：阻塞和非阻塞强调的是应用层等待数据返回时的线程状态问题，比如当前线程是挂起等待，还是可以抽身去干其他的事情。

BIO：同步阻塞I/O模型，阻塞主要体现在socket的接收/读/写函数都是同步阻塞的。如果当前没有连接进来，会在serverScoket.accept()函数上阻塞;如果当前连接没发送数据，服务端就会在clientSocket.getInputStream()上阻塞；（现实场景：考驾照，必须本人去考，且全程都是同步阻塞的）

NIO：同步非阻塞I/O模型。非阻塞主要体现在socket的接收/读/写函数上，在等待I/O就绪阶段都是非阻塞的，主要通过I/O多路复用技术来实现。（现实场景：排队点餐，排队的时候自己并不是只能傻傻的等着，还可以做其他的事情，比如看我的Tomcat源码系列视频，比如看个小电影，玩个推塔游戏等）

AIO（NIO2）：异步非阻塞I/O模型。异步主要体现I/O的读写操作都是异步的，它不是在IO准备好时再通知线程，而是在IO操作已经完成后，再给线程发出通知。因此AIO是不会阻塞的，此时我们的业务逻辑将变成一个回调函数，等待IO操作完成后，由系统自动触发。（现实场景：12306后补票，不需要一直盯着候补状态，如果成功或失败App都会短信通知，这一过程完全可以全身心干其他事情）

BIO NIO AIO

基于流（输入流InputStream和输出流OutputStream） 基于缓存区（Buffer，Channel双向通道，可读可写） 基于回调函数
无选择器 拥有选择器Selector（I/O多路复用器） 无选择器 
1.阻塞IO（BIO） BIO是Java最初的IO模型，它采用阻塞方式进行数据读写操作。即当一个线程在执行IO操作时，若没有数据可读，则该线程会一直阻塞等待，直到有数据可读或者超时。BIO适合处理连接数比较小且固定的场景，但并发能力不足。
2.非阻塞IO（NIO） NIO是Java 1.4引入的新的IO模型，它采用了多路复用器（Selector）机制，通过少量线程同时管理多个通道，实现了单线程同时处理多个请求的效果。NIO具有高并发性、高吞吐量和更高的可靠性，适合处理连接数多且连接时间较短的场景。
3.异步IO（AIO） AIO是Java 1.7开始支持的IO模型，它采用事件驱动的方式进行数据读写操作，当数据准备好后，在回调函数中进行处理。与NIO不同，AIO的读写操作是异步的，不需要通过轮询方式去检查数据是否准备好。AIO适合处理连接数多、连接时间长且有较多读写操作的场景。

public void init() throws Exception {
   /**初始化/实例化类加载器（common、catalina、shared），当catalina和shared未指定时，这三个指向同一个*/
   initClassLoaders();
   /**
    * 设置当前线程上下文类加载器为catalinaLoader即server类加载器
    * 主要为了打破父委托模式，让父类加载器加载到子类加载器路径下的class
    * ①Tomcat中的代码应用场景：{@link Digester#getClassLoader()}
    * ②其他应用场景：
    * 如果有N个Web应用程序都是用Spring来进行组织和管理的话,
    * 可以把Spring依赖放到Shared目录下让这些程序共享。Spring要对应用程序的类进行管理，
    * 自然要能访问到应用程序的类，而应用程序的类显然是放
    * 在/WebApp/WEB-INF目录中的，那么被SharedClassLoader加载的Spring如何访问并不在
    * 其加载范围内的应用程序类呢？ 那就只能用下面这种方式了，这就打破了父委派模式，还有其他：JNDI、JDBC等都是
   */
   Thread.currentThread().setContextClassLoader(catalinaLoader);
   Class<?> startupClass = catalinaLoader.loadClass("org.apache.catalina.startup.Catalina");
   /**
    * 利用反射实例化Catalina对象
    * 这里为什么要使用反射？ 你看源码里Bootstrap和Catalina就差“睡在一起了”，为什么还要通过反射来实例化呢？
    * 答案在上一小节，可以自己再重温下
    */
   Object startupInstance = startupClass.getConstructor().newInstance();
   String methodName = "setParentClassLoader";
   Class<?> paramTypes[] = new Class[1];
   /**参数类型是：ClassLoader*/
   paramTypes[0] = Class.forName("java.lang.ClassLoader");
   /**参数个数1个，其实就是共享的类加载器*/
   Object paramValues[] = new Object[1];
   /**
     * 这个地方注意了，整个Catalina下面所有的容器都将使用这个sharedLoader类加载实例作为父
     * 比如StandardContext中的{@link org.apache.catalina.loader.WebappClassLoaderBase}的父加载器就是sharedLoader
     */
   paramValues[0] = sharedLoader;
   Method method = startupClass.getMethod(methodName, paramTypes);
   /**设置Catalina容器下面的所有组件的父类加载器为shareLoader*/
   method.invoke(startupInstance, paramValues);
   /**设置catalina实例的ref*/
   catalinaDaemon = startupInstance;
}

8.3.3 createClassLoader

/**
  * 创建类加载器
  * @param name 加载所指定的配置名称
  * @param parent 上一层类加载器
  * @return 如果name没有配置，那就直接返回parent，对于server和shared来说，父类加载器就是CommonLoader
  * @throws Exception
  */
private ClassLoader createClassLoader(String name, ClassLoader parent) throws Exception {
        /**
         * 在catalina.properties中有三个name.loader选项，如下：
         * common.loader="${catalina.base}/lib","${catalina.base}/lib/*.jar","${catalina.home}/lib","${catalina.home}/lib/*.jar"
         * server.loader= , 这个server.loader没有值，tomcat作者说了，如果这个留空，那server的类加载器就是commonLoader
         * (以Common类加载器为父类，适用于加载Tomcat自身依赖的Jar的类加载器，其路径由server.loader指定，默认为空，此时tomcat使用Common类加载器加载自身所需的jar。)
         * shared.loader= , 这个shared.loader没有值，tomcat作者说了，如果这个留空，那shared的类加载器就是commonLoader
         * 以Common类加载器为父类，是所有Web应用的父类加载器，其路径由shared.loader指定，默认为空，此时tomcat使用Common类加载器作为Web应用的父加载器。
         *
         * Web应用：以Shared类加载器为父类，加载/WEB-INF/classes目录下的未压缩的Class和资源文件以及/WEB-INF/lib目录下的jar包，
         *         该类加载器只对当前Web应用可见，对其他Web应用均不可见。
         * 默认情况下，Common、Catalina、Shared类加载器是同一个，但可以配置3个不同的类加载器，使他们各司其职
         */
        String value = CatalinaProperties.getProperty(name + ".loader");
        /**如果找不到name对应的loader配置选项的话，就返回父类加载器*/
        if ((value == null) || (value.equals(""))) {
            return parent;
        }

        /**value值是一个带变量表达式的字符串，如 "xxx/${catalina.base}/lib",replace就是要替换变量换成具体的值 */
        value = replace(value);

        /**Repository是ClassLoaderFactory 中的一个静态内部类, 有2个属性, location, type, 表示某个位置的某种类型的文件*/
        List<Repository> repositories = new ArrayList<>();

        /**
         * 默认情况下，如果我们不配置catalina.properties中的common.loader标签的值的话，这里取出的数组就4个
         * 分别是：
         * 1、${catalina.base}/lib
         * 2、${catalina.base}/lib/*.jar
         * 3、${catalina.home}/lib
         * 4、${catalina.home}/lib/*.jar
         *
         * bin (运行脚本）
         * conf (配置文件）
         * lib (核心库文件）
         * logs (日志目录)
         * temp (临时目录)
         * webapps (自动装载的应用程序的目录）
         * work (JVM临时文件目录[java.io.tmpdir])
         * 其中只有bin和lib被多个tomcat示例共用，其它目录conf、logs、temp、webapps和work 每个Tomcat实例都拥有独自一份
         * 所以：
         *
         * catalina.home(安装目录)：指向公用信息的位置，就是bin和lib的父目录。
         * catalina.base(工作目录)：指向每个Tomcat目录私有信息的位置，就是conf、logs、temp、webapps和work的父目录。
         * 在只有一个tomcat实例的情况下，home和base是一样的
         * */
        String[] repositoryPaths = getPaths(value);

        for (String repository : repositoryPaths) {
            // Check for a JAR URL repository
            try {
                @SuppressWarnings("unused")
                /**
                 * 一上来先暴力检查是否是一个url类型的仓库，如果不是，
                 * tomcat原来异常是忽略不处理，后面会再判断是不是glob、jar或者dir类型的
                 * 类型是一个枚举类，有：
                 * DIR, -- 目录格式的，如 file://xxxxxx
                 * GLOB, -- N个jar，*.jar结尾的都算
                 * JAR, -- 单个jar
                 * URL -- 网络地址形式的，如http://xxxxxx
                 */
                URL url = new URL(repository);
                repositories.add(new Repository(repository, RepositoryType.URL));
                continue;
            } catch (MalformedURLException e) {
                // Ignore
                System.out.printf("%s - 不是URL repository \n",repository);
            }

            // Local repository , 既然不是网络的，那一定是本地文件系统可以访问到的
            if (repository.endsWith("*.jar")) {
                repository = repository.substring
                    (0, repository.length() - "*.jar".length());
                repositories.add(new Repository(repository, RepositoryType.GLOB));
            } else if (repository.endsWith(".jar")) {
                repositories.add(new Repository(repository, RepositoryType.JAR));
            } else {
                repositories.add(new Repository(repository, RepositoryType.DIR));
            }
        }

        return ClassLoaderFactory.createClassLoader(repositories, parent);
}

9.1.1 什么是Digester

​ 一款非常强大的将Xml文件配置内容转换为Java对象的解析框架，隶属于ASF旗下，Tomcat作者只不过是将其源码直接拷贝到了项目中，然后对其进行了改造，如增加了很多节点解析的规则类等。

​ 官方对它的描述是：

The Digester package lets you configure an XML->Java object mapping module which triggers certain actions called rules whenever a particular pattern of nested XML elements is recognized.

​ Digester最初是作为Struct MVC框架的一个工具模块，来完成xml解析的功能。但是很快地，有人发现并觉得Digester不应该仅仅局限在Struct，而应该变得更通用。于是经过Apache的孵化，最终加入到了Apache Commons组件库家族中，并形成了一个功能强悍的XML解析库。

​ Digester底层是基于SAX+事件驱动+栈的方式来搭建实现的。那么在Digester中，这三种元素分别起到什么作用呢？

1. SAX，用于解析xml；
2. 事件驱动，在SAX解析的过程中加入事件来支持我们的对象映射;
3. 栈，当解析xml元素的开始和结束的时候，需要通过xml元素映射的类对象的入栈和出栈来完成事件的调用;

​ Digester解析xml的核心流程主要可以概况如下4个步骤：

1. 开始解析某个节点的时候，是否需要创建一个类
2. 开始解析某个节点的时候，是否需要入栈操作
3. 结束解析某个节点的时候，是否需要执行某个方法
4. 结束解析某个节点的时候，是否需要出栈操作

9.1.2 什么是SAX

​ 定义：SAX(Simple API for XML) XML简单应用程序接口。SAX提供的访问模式是一种基于文件流的顺序模式，是一种快速读写XML数据的方式。当使用SAX分析器对XML文档进行分析时，会触发一系列的事件，并激活相应的事件处理函数，应用程序通过这些事件处理函数实现对XML文档的访问，因而SAX接口也被称件事件驱动接口
​ 解析XML文档方式有两种，一种是DOM，另外一种就是SAX：

DOM--解析XML时，首先要将XML文档加载到内存当中，然后可以通过随机访问方式访问内存中的DOM树，访问大型xml文档时会出现内存溢出的错误。

SAX--解析XML时，是基于事件而且是顺序执行的，解析的时候随时可以停止，且一旦解析过了某个元素，就没有办法再去访问它了。SAX不必将整个XML文档都加载到内存中，因此占用的内存要比DOM小，对于大型的XML文档来说，通常会使用SAX而不是DOM进行解析。

常用接口介绍

  • ContentHandler
ContentHandler是Java类包中一个特殊的SAX接口，位于org.xml.sax包中。该接口封装了一些对事件处理的方法，当XML解析器开始解析XML输入文档时，它会遇到某些特殊的事件，比如文档的开头和结束、元素开头和结束、以及元素中的字符数据等事件。当遇到这些事件时，XML解析器会调用ContentHandler接口中相应的方法来响应该事件。
  • DTDHandler
DTDHandler用于接收基本的DTD相关事件的通知。该接口位于org.xml.sax包中。此接口仅包括DTD事件的注释和未解析的实体声明部分。SAX解析器可按任何顺序报告这些事件，而不管声明注释和未解析实体时所采用的顺序；但是，必须在文档处理程序的startDocument()事件之后，在第一个startElement()事件之前报告所有的DTD事件。
  • EntityResolver
EntityResolver接口是用于解析实体的基本接口，该接口位于org.xml.sax包中。
  • ErrorHandler
ErrorHandler接口是SAX错误处理程序的基本接口。如果SAX应用程序需要实现自定义的错误处理，则它必须实现此接口，然后解析器将通过此接口报告所有的错误和警告。

核心默认Handler实现DefaultHandler

DefaultHandler类是SAX2事件处理程序的默认基类。它实现了EntityResolver、DTDHandler、ContentHandler和ErrorHandler这四个接口。包含这四个接口的所有方法，所以我们在编写事件处理程序时，可以不用直接实现这四个接口，而继承该类，然后重写我们需要的方法

当然在Tomcat源码中，Digester继承的是DefaultHandler2类，其扩展了一些方法，这个就不多说了！

9.2 Connector

​ 连接器用于处理连接和并发，内部屏蔽了不同应用层协议和网络IO模型，网络IO模型与应用层协议直接可以任意组合，这种组合的抽象封装对应的核心接口为ProtocolHandler（位于Coyote模块中）

​ Tomcat支持的应用层协议有：

	* ①HTTP1.1 
	* ②HTTP2（性能非常优越，浏览器请求并发数不受限制） 
	* ③AJP（Apache JServ Protocol，定向包协议，通过二进制进行传输而不是文本，因此性能高）；

​ Tomcat支持的网络IO模型有：

	* ①BIO（同步阻塞，Tomcat8.5之后已经移除掉了）
	* ②NIO（同步非阻塞，基于JDK的NIO库实现，Tomcat内部实现了自己的Reactor模型，即单Reactor多线程，不要再说为什么Tomcat不用Netty了，因为Tomcat自己实现了啊）
	* ③AIO（异步非阻塞，体现在Tomcat中就是NIO2，NIO2和AIO是一回事，基于JDK的NIO2库实现）
	* ④APR（使用C++本地库实现，从操作系统层面来处理异步式IO，性能较高，但是需要安装依赖）。

     * 容器表单 URL 参数解析将处理的 POST 的最大大小（以字节为单位）。通过将此属性设置为小于零的值，
     * 可以禁用该限制。如果未指定，则此属性设置为 2097152(2 MB)。请注意，FailedRequestFilter 可用于拒绝超过此限制的请求。
     */

     * Coyote adapter.
     * coyote适配器，负责将前端的request请求转换成HttpServletRequest转交给container（就是servlet应用，后端服务）

        boolean apr = AprStatus.getUseAprConnector() && AprStatus.isInstanceCreated()
                && AprLifecycleListener.isAprAvailable();
        ProtocolHandler p = null;
        try {
            /**基于协议类型和是否开启apr来创建协议处理器*/
            p = ProtocolHandler.create(protocol, apr);
        } catch (Exception e) {
            log.error(sm.getString(
                    "coyoteConnector.protocolHandlerInstantiationFailed"), e);
        }
        if (p != null) {
            protocolHandler = p;
            protocolHandlerClassName = protocolHandler.getClass().getName();
        } else {
            protocolHandler = null;
            protocolHandlerClassName = protocol;
        }
        // Default for Connector depends on this system property
        setThrowOnFailure(Boolean.getBoolean("org.apache.catalina.startup.EXIT_ON_INIT_FAILURE"));
    }

            return false;
        }
        return IntrospectionUtils.setProperty(protocolHandler, name, value);

第10章 Tomcat Coyote

​ Coyote英译为郊狼，是一种产于北美大草原的小狼，但在Tomcat源码中实则是一个模块的名称，这个模块统称为连接器，在第9章中我们提到了Catalina模块中的Connector，它虽然直译也叫连接器，但它代表的层面只能是单一的类，其以组合的方式将socket和http的处理部分进行了很好的包装（串联），而我们今天要说的Coyote可以说是Tomcat Connector框架非常核心的一个模块，甚至直接说它就是Tomcat的连接器也不为过。

​ 不论Tomcat的容器设计得多么巧妙，其本质上它就是一个Http服务器，主要就是从socket中解析获取http数据流，然后将http数据流封装成ServletRequest和ServletResponse传给遵循servletN.x的web应用，由web应用相应的servlet类处理完成再后将结果写入到ServletResponse中的一个过程，这一过程核心的就是socket对象和request对象之间的转换，这么重要的功能在Tomcat源码中实现的部分就是我们此次要说的Coyote。

10.1 ProtocolHandler

协议处理器

10.2 Adapter

connector与container之间网络信息（请求与响应）通信的适配器。

10.3 AbstractEndpoint

通信端点，主要就是socket套接字（高性能nio，IO多路复用），监听网络连接、处理客户端socketChannel，然后注册对应的感兴趣事件，Poller线程轮询遍历事件，最后交给对应协议的Processor处理。

10.4 Processor

SocketChannel（客户端套接字通道，该通道即可读也可以写）处理器，主要体现在默认的Http11Processor（其内含Coyote模块的Request和Response实例，同时还含有Catalina模块中的CoyoteAdapter实例对象）的service(SocketWrapperBase<?> socketWrapper)方法中。

10.5 Request

不同于Catalina模块中的Request（其内部包含了coyote的Request，且Catalina的Request遵循Servlet规范，实现了HttpServletRequest接口），这个是Coyote模块中的Request，更加原生，更加贴近于客户端套接字（因为其是Http11Processor对客户端套接字通道对象解析后进行request相关属性赋值的），其最终会被Catalina模块中的Request对象引用！

10.6 Response

同上述阐述的Request。

12.1.1 JNDI是什么

​ JNDI（Java Naming and Directory Interface,Java命名和目录接口）在J2EE规范中是重要的规范之一。是一组在Java应用中访问命名和目录服务的API，是SUN公司提供的一种标准的Java命名系统接口，命名服务将名称和对象联系起来，使得我们可以用名称访问对象。目录服务是一种命名服务，在这种服务里，对象不但有名称，还有属性。

​ JNDI可访问的现有的目录及服务有：DNS、XNam 、Novell目录服务、LDAP(Lightweight Directory Access Protocol 轻型目录访问协议)、 CORBA对象服务、文件系统、Windows XP/2000/NT/Me/9x的注册表、RMI、DSML v1&v2、NIS。

​ JNDI作用：

  • JNDI避免了程序与数据库之间的紧耦合，使应用更加易于配置、易于部署。
  • 所有与系统外部的资源的引用，都可以通过JNDI定义和引用。

​ JNDI的优点：

* 包含了大量的命名和目录服务，使用通用接口来访问不同种类的服务；
* 可以同时连接到多个命名或目录服务上；
* 建立起逻辑关联，允许把名称同Java对象或资源关联起来，而不必指导对象或资源的物理ID。

12.1.2 白话JNDI

​ 上面说了一大堆，没用过的人其实可能还不明白，甚至只知道这个概念很牛逼，但是用在什么场景下却一脸懵逼！？（如果你理解SPI并用过，就知道JNDI是什么了）我这里举一个例子，后面我还会结合一个具体的小案例来演示它是怎么用的。

		例子：在没有出现Mybatis/Spring/SpringBoot等之前（"远古时期"企业级Java服务开发用的都是EJB和JNDI等规范），我们要想访问DB，必须使用JDBC那一套编码风格，如下：

6、处理结果：rowMapper.rowMapper(resultSet);
7、释放资源：finally{xxx.close();xxx.close();....}

​ 上面有个弊端，就是数据源的配置全部都是写死在代码里的，不通用维护起来也麻烦，因为配置这玩意就是易变动的，开发和部署环境都是不一样的，怎么办？那就不写死呗？ 不写死，那放到哪呢？提出来啊？提到哪呢？当然是提到配置文件里啊，比如*.xml 里啊。那我怎么读取.xml里的内容呢？…..

​ 说到读了，就不得不提一下，这里有几个角色：

​ ① J2EE（早期是Sun，现在是Oracle JavaEE，开源的是JakartaEE）

​ ② 服务供应商

​ ③ 用户

​ 其中J2EE定义规范，比如你得给这个配置起一个名称（naming），好让我们（用户）在用的时候通过名称可以快速找到对应的配置上下文实例，然后上下文实例中的属性你（服务供应商）要可以列出来（目录）并可以获取。具体怎么列和获取，我J2EE不管，我只管定义好接口，实现由各个服务供应商来，比如数据库驱动供应商：Oracle、PostgreSQL、MySql等。

@WebServlet(name="usersServlet",value = "/user/list")
